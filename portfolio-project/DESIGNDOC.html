Portfolio Webpage Design Doc

Setup

Various coding demonstration projects need an appropriate showcase which itself doubles as another project.

Problem

A demonstration website requires certain high functionality standards regarding content, layout, image preloading, form validation, project data presentation, and user accessibility. The system is more than the sum of its parts, and each part must integrate into the whole seamlessly.

Goals

  • Display software programming, organization and planning, and system design competency
  • Showcase an engaging, dynamic frontend website
  • Make data presentation a subtle, pleasurable experience
  • Create GUIs that are user friendly and intuitive
  • Make system easy to use and hard to break
  • Optimize time management

Options

Script Loading and Execution

  1. None: JS loads and executes immediately, blocking parsing until the script is completed
  2. Defer: JS loads in parallel with HTML parsing and executes after parsing is complete [current]
  3. Async: JS loads in parallel with HTML parsing and executes as soon as available

Content Organization

  1. Include a single index HTML projects page with no additonal HTML or external hyperlinks required
  2. Include an index HTML homepage with external hyperlinks to projects, github, resume, faq, and contacts
  3. Include an index HTML homepage; additional HTML pages for projects, faq, and contacts; and external hyperlinks to github and resume [current]

Content Storage

  1. Use website as static silo storage, and include all content within the website server folder
  2. Use website as dynamic presentation hub, and link to external content whenever possible [current]

Layout Artistic Themes

  1. Do not use any artistic themes
  2. Use prefabricated static CSS
  3. Use prefabricated keyframe animated CSS
  4. Create original static CSS [current]
  5. Create original keyframe animated CSS

Layout Organization

  1. No CSS, let the HTML fall where it will
  2. Use basic CSS
  3. Use CSS Flexbox
  4. Use CSS Grid [current]

Layout Scaling

  1. Use single static layout regardless of screen size
  2. Use media queries with breakpoints for large screens and phones
  3. Use media queries with breakpoints for large screens, tablets, and phones [current]
  4. Use media queries with breakpoints for hi res screens, large screens, tablets, and phones

Image Preloading

  1. Do nothing and let image icons sit onscreen until all the images intitially load
  2. List sources for the other more image heavy pages on initial image single image homepage using JS [current]>
  3. Use a window onload global event handler to forestall page updating while the images load
  4. Play a skeleton loading animation while page images load

Form Validation

  1. Have neither form validation nor error messages
  2. Use automatic HTML "required" client side validation and error messages
  3. Use custom JS client side validation and error messages [current]
  4. Use custom JS.node server side validation and error messages
  5. Use custom PHP server side validation and error messages

Project Data Presentation

  1. Use separate HTML webpages for each project, but do not use any JS to jazz it up
  2. Use separate HTML webpages for each project, then use subtle JS touches to jazz it up
  3. Use separate HTML webpages for each project, then use flamboyant JS to overwhelm the viewer
  4. Use a single page application HTML to present all projects at once, but do not use any JS to jazz it up
  5. Use a single page application HTML to present all projects at once, then use subtle JS touches to jazz it up [current]
  6. Use a single page application HTML to present all projects at once, then use flamboyant JS to overwhelm the viewer

User Accessibility

  1. Do not implement any user accessibility techniques
  2. Use high foreground/background contrast
  3. Use high foreground/background contrast and image alt tags [current]
  4. Use high foreground/background contrast, image alt tags, and keyboard shortcuts
  5. Use high foreground/background contrast, image alt tags, keyboard shortcuts, and dynamic settings selection options

Solution

Demonstrate core competencies without too many bells and whistes to distract from the clean, minimal requirements of this website: present the site creator as an innovative software designer and manager by highlighting and glorifying his or her web application development accomplishments. All else is dross. Allow for multiple target users—programmers vs managers or the logical vs the artistic—to easily select and view what data they prefer—either code, design docs, or working prototypes—using an organized, intuitive layout. Keep peripherals such as resumes and githubpages behind external links. The github.io static hosting restrictions and lack of php or node.js experience will limit the backend applications of this website, so polish the frontend and warn users that they can't actually send emails, etc.
