Executive Summary

The purpose of this project is to display the software documentation, design philosophy, developmental skills, and managerial acumen of its creator Jeffrey Bardwell by means of a website built using largely vanilla HTML, CSS, and JS code with only a few libraries and no frameworks. That choice is made as a compromise between overreliance on cookie cutter code and creating everything from scratch and to demonstrate the creator's versatility as a problem solver. After all, the site code is as much on display as the site content. That content acts as a platform for generating interest in the creator's brand name as well as showcasing his various software projects. The site successfully demonstrates instances of static, dynamic, and data driven code. Design is presented in terms of the mechanics-dynamics-aesthetics paradigm.

This is a front end website whose aim is to present and summarize data rather than collect or send it, eschewing back end functionality. This option was selected as the best way to manage creator time and skill, with several features being wishlisted in the interests of expedient project completion. Some wishlisted features have been implemented, others not. This website does not have all the bells and whistles the creator would have wanted, but one must know when to step away, dust one's hands after so many iterations, and say job done: adequate pending on marvelous. One will never truly reach marvelous any more than a soul will ever truly reach nirvana. Perfectionism is self-defeating and self-destructive. Meanwhile, the next project awaits, twiddling its thumbs. Pause. Debrief. Move on. That dash of ice water aside, the minimum and core project features of this project have all been defined, iterated, tested, and implemented. Site development is dynamic and ongoing as are the projects whose data it presents. The concept is sound; the foundation is strong, the pieces are modular. The website is itself showcased as one of those projects on the website with a progress report detailing relevant completion metrics.

To summarize those metrics, most of the eight primary project goals are fully met, with two exceptions that are only partially satisfied. Of the different file types created and iterated, HTML (73%) is the most prominent, followed by JS (24%), and then CSS (4%). These numbers are somewhat skewed as many HTML documents made to be inserted into other HTML documents have inline or internal CSS code, and those percentages only account for external code CSS files. In terms of relative time spent, as a one-man team project, documentation, design, and development activities heavily outpace managerial activities. However, all tangible and intangible outcomes and skills demonstrated in this project are scaleable and adaptable with the potential to suit the needs of many different hypothetical team configurations.


Fundamentals

Setup

Several software demonstration projects require an online showcase, which itself doubles as another software demonstration project.


Problem

A frontend demonstration website requires meeting—and in the best-case scenario wildly exceeding—certain standards regarding documentation, design, content, layout, image preloading, form validation, project data presentation, and user accessibility. Aesthetics and system design considerations must be balanced with practical and elegant code. Across the hierarchy of mechanics, dynamics, and aesthetics, a creation is judged by the user experiences it generates and those impressions reflect upon the one who created it.


Solutions

Website Build Options

Option Pros Cons
1a. Use a styleguide template Prefabricated styleguides are ubiquitous. Demonstrates proficiency in styleguide aesthetics. Does not offer the opportunity to demonstrate styleguide creation proficiency and that template may be a pain to modify.
1b. Create an original styleguide Demonstrates proficiency in styleguide creation and organization. Someone—likely several someones each with more artistic flair and technical skill than you—has already written and tested the code to do what you require.
2a. Use common framework(s) Presupposing foreknowledge, this demonstrates prowess in the particular framework(s) used and can generate code quickly. This code is rigid and difficult to modify. Code generation may be swift but debugging is not. Code is bloated, verbose, and dependency ridden with copious workarounds and exceptions.
2b. Use vanilla code in lieu of framework(s) Demonstrates code architecture competency. Allows for freedom and flexibility when designing and creating system mechanics. Someone—likely several someones each more competent and skilled than you—has already written and tested the code to do what you require.
3a. Use copious amounts of libraries Why reinvent the wheel? Takes advantage of pre-existing code and modules to complete a given task. This wheel probably doesn't fit your specific make or model of vehicle and must be pummeled with a tire iron until it "fits" to spec.
3b. Use libraries sparingly A compromise between comparison shopping for a shirt to fit your frame in a store versus gathering the materials, measuring the fabric, and stitching the garment from whole cloth. Like most middle of the road options, there are no great disadvantages other than those caused by trying to mutally satisfy two opposing choices.
3c. Use pure vanilla code in lieu of libraries This is the best method for demonstrating code architecture and overall fundamental proficiency. Are you masochistic? This requires a full team engaging in copious amounts of writing and debugging. Unless extenuating circumstances warrant, it's an egregious waste of time!

Selection Rationale

Of the 12 different plan configrations presented by the options above, I selected suboptions 1b, 2b, and 3b to strike a balance between churning out cookie cutter code and having to create everything from scratch. Sadly, this did not allow me to showcase my nonexistant framework skills. Frameworks are no more than scaffolding upon a foundation, and if the foundation is weak, what use conveniently erected timbers? So, I played to my strengths and chose a plan that lets me flex my HTML, CSS, and JS muscles. This allows me to demonstrate my system design as well as coding prowess. It also provides more options for customization; is broadly applicable to a wider range of employment opportunities; and showcases my grasp of the fundamentals planning, writing, testing my own code mechanics while still making judicious use of a few handy libraries.

Aesthetics

Logo Design

I wanted an image that screamed "internet," so the wireframe world topography with revolving arrow and san serif typography reflect that. Other than minor modifications such as the golden backdrop and imperial purple palette upgrade, the stock image is the original royalty free version. The colors purple and gold harken to something imposing, decadent, and both possessing and requiring a wealth of resources. The moonglow effect is both thematically appropriate for the starry night webpage motif and helps the image pop against a dark background whilst vanishing entirely into a white background.

Site Design

This website needed to be simple, evocative, and easy to navigate. The starry night motif connotes dreams, aspiration, and calm. The navigation menu is placed within the tall grass silhouette to reinforce the perspective of one looking up at the limitless promise dangling above them waiting to be grabbed: the wonder, the majesty, the dream. Thus, the menu—portal to those pesky details of actualizing that dream—is more down to earth.

This orientation, not so coincidentally, also centers my name in the heavens, which I thought was a nice touch. The random quotes are from a selection of my favorite folks on topics ranging from imagination, philosophy, stars, systems, vocation, to writing.


Definitions

  • data-driven: software whose structure and content are both softcoded using dom creation and data or databases, respectively

  • dynamic: software whose structure is hardcoded but content is softcoded and subject to dom manipulation

  • static: software whose structure and content are both hardcoded and unchanging
