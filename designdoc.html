Setup

Several software demonstration projects require an online showcase, which itself doubles as another software demonstration project.


Problem

A frontend demonstration website requires meeting—and in the best-case scenario wildly exceeding—certain standards regarding documentation, design, content, layout, image preloading, form validation, project data presentation, and user accessibility. Aesthetics and system design considerations must be balanced with practical and elegant code. Across the hierarchy of mechanics, dynamics, and aesthetics, a creation is judged by the user experiences it generates and those impressions reflect upon the one who created it.


Solutions

Website Build Options

Option Pros Cons
1a. Use a styleguide template Prefabricated styleguides are ubiquitous. Demonstrates proficiency in styleguide aesthetics. Does not offer the opportunity to demonstrate styleguide creation proficiency and that template may be a pain to modify.
1b. Create an original styleguide Demonstrates proficiency in styleguide creation and organization. Someone—likely several someones each with more artistic flair and technical skill than you—has already written and tested the code to do what you require.
2a. Use common framework(s) Presupposing foreknowledge, this demonstrates prowess in the particular framework(s) used and can generate code quickly. This code is rigid and difficult to modify. Code generation may be swift but debugging is not. Code is bloated, verbose, and dependency ridden with copious workarounds and exceptions.
2b. Use vanilla code in lieu of framework(s) Demonstrates code architecture competency. Allows for freedom and flexibility when designing and creating system mechanics. Someone—likely several someones each more competent and skilled than you—has already written and tested the code to do what you require.
3a. Use copious amounts of libraries Why reinvent the wheel? Takes advantage of pre-existing code and modules to complete a given task. This wheel probably doesn't fit your specific make or model of vehicle and must be pummeled with a tire iron until it "fits" to spec.
3b. Use libraries sparingly A compromise between comparison shopping for a shirt to fit your frame in a store versus gathering the materials, measuring the fabric, and stitching the garment from whole cloth. Like most middle of the road options, there are no great disadvantages other than those caused by trying to mutally satisfy two opposing choices.
3c. Use pure vanilla code in lieu of libraries This is the best method for demonstrating code architecture and overall fundamental proficiency. Are you masochistic? This requires a full team engaging in copious amounts of writing and debugging. Unless extenuating circumstances warrant, it's an egregious waste of time!

Selection Rationale

Of the 12 different plan configrations presented by the options above, I selected suboptions 1b, 2b, and 3b to strike a balance between churning out cookie cutter code and having to create everything from scratch. Sadly, this did not allow me to showcase my nonexistant framework skills. Frameworks are no more than scaffolding upon a foundation, and if the foundation is weak, what use conveniently erected timbers? So, I played to my strengths and chose a plan that lets me flex my HTML, CSS, and JS muscles. This allows me to demonstrate my system design as well as coding prowess. It also provides more options for customization; is broadly applicable to a wider range of employment opportunities; and showcases my grasp of the fundamentals planning, writing, testing my own code mechanics while still making judicious use of a few handy libraries.

Implementation Options

Various options are categorized by common functionality and arranged in ascending order of complexity. Currently implemented options are bracket labeled as such and highlighted with bold text. These selections are always updated to reflect current project status.
